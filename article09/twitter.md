# День 9 цикла «LLM‑агенты для трейдинга»: Масштабирование агентов

## Вариант 1 (простой тред)

🚀 День 9: Масштабирование агентов — управление множеством пар, базы данных и панели мониторинга

Один агент на одну пару — это только начало. Пора строить систему, которая управляет десятками активов, хранит историю и сама восстанавливается после сбоев.

🧵 Ниже — что вы узнаете из статьи:

---

1/12
🏗️ Архитектура масштабированной системы: агенты‑воркеры, диспетчер, база данных, панель мониторинга, оркестратор, логирование.

---

2/12
👷 Мультипроцессный мониторинг: запускаем десятки воркеров через `multiprocessing`. Каждый процесс изолирован, падение одного не влияет на остальных.

---

3/12
⚙️ Менеджер процессов: класс `ProcessManager`, который стартует, останавливает и перезапускает воркеров на основе YAML/JSON‑конфигурации.

---

4/12
🗄️ База данных SQLite: таблицы для свечей, индикаторов, сигналов, ордеров и снимков портфеля. Функции `save_candle()`, `save_signal()`.

---

5/12
📊 Панель мониторинга (вариант A): Prometheus + Grafana. Экспортируем метрики (цена, RSI, сигналы) через `prometheus_client`.

---

6/12
📊 Панель мониторинга (вариант B): простой Flask‑интерфейс с JSON API (`/api/prices`, `/api/signals`) и HTML‑дашбордом.

---

7/12
🐕‍🦺 Оркестрация: systemd службы для каждого компонента (`trading-manager.service`, `dashboard.service`). Конфигурационные файлы.

---

8/12
🔄 Watchdog: скрипт, который каждую минуту проверяет статус служб через `systemctl is‑active` и перезапускает упавшие, отправляя уведомление в Telegram.

---

9/12
🔧 Интеграция с OpenClaw: обновляем skill `binance‑trading`, добавляем новые скрипты (`worker.py`, `process_manager.py`, `database.py` и др.).

---

10/12
🧪 Тестирование: поэтапный запуск — инициализация БД, менеджер, панель, watchdog. Тестируем на демо‑аккаунте с 2‑3 парами.

---

11/12
🚀 Переход на production: замена SQLite на PostgreSQL, Docker‑контейнеры, бэкапы БД, SSL для веб‑интерфейса, тонкие алерты.

---
