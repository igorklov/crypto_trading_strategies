## 4. Проблемы, которые мы обнаружили

Несмотря на тысячи сигналов, реальные ордера почти не исполнялись. Анализ логов системы выявил несколько критических проблем:

### 4.1. Ошибка округления количества (`LOT_SIZE`)

Binance требует, чтобы количество актива в ордере соответствовало шагу `stepSize`, определённому фильтром `LOT_SIZE`. Наша система рассчитывала количество как `free_balance * trade_fraction`, что давало значения с избыточной точностью (например, `1013.3000000000001`). API Binance возвращал ошибку:

```
Parameter 'quantity' has too much precision.
```

**Решение:** необходимо округлять количество до допустимого шага, используя информацию о фильтрах символа, которую можно получить через `exchangeInfo`.

### 4.2. Недостаточный баланс на демо‑счете

Демо‑счет Binance Testnet изначально содержит лишь небольшое количество тестовых USDT и нулевые балансы по другим активам. При попытке продажи актива, которого нет на балансе, возникала ошибка:

```
Account has insufficient balance for requested action.
```

**Решение:** перед запуском системы нужно пополнить демо‑счет тестовыми активами через faucet Binance Testnet или реализовать механизм начального депозита.

### 4.3. Игнорирование фильтра `NOTIONAL`

Даже если количество округлено правильно, ордер может быть отклонён из‑за нарушения минимального номинала (`minNotional`). Наша система не проверяла этот фильтр для всех пар.

**Решение:** добавить проверку `quantity * price >= minNotional` и при необходимости увеличивать количество до минимального порога.

### 4.4. Отсутствие записи ордеров в базу данных

Хотя ордера размещались через API Binance, их результаты не сохранялись в локальную базу, что затрудняло последующий анализ P&L.

**Решение:** расширить таблицу `orders` и записывать в неё каждый размещённый ордер (ID, сторона, цена, количество, статус, комиссия).

## 5. Как мы исправили ошибки (практические шаги)

В процессе отладки мы внесли следующие изменения в код воркера (`worker.py`):

### 5.1. Получение фильтров символа

```python
def _get_symbol_info_filters(self):
    """Получает фильтры LOT_SIZE и NOTIONAL для символа."""
    try:
        info = self.trader.client.get_symbol_info(self.symbol)
        filters = info['filters']
        min_qty = None
        step = None
        min_notional = None
        for f in filters:
            if f['filterType'] == 'LOT_SIZE':
                min_qty = float(f['minQty'])
                step = float(f['stepSize'])
            elif f['filterType'] == 'NOTIONAL':
                min_notional = float(f['minNotional'])
        return min_qty, step, min_notional
    except Exception as e:
        print(f"Error fetching symbol filters: {e}")
        return None, None, None
```

### 5.2. Округление до шага LOT_SIZE

```python
def _round_to_step(self, quantity, step):
    """Округляет количество до шага stepSize."""
    if step <= 0:
        return quantity
    # Определяем количество знаков после запятой для step
    decimals = int(round(-math.log10(step)))
    # Округляем до этих знаков
    return round(math.floor(quantity / step) * step, decimals)
```

## 6. Итоги торговли: что получилось, а что — нет

### Успехи:
- Система стабильно работала 4 дня без перерывов.
- Собрана богатая база данных: более 9000 сигналов, свечи, индикаторы.

### Неудачи:
- Из‑за ошибок округления и недостатка баланса ни один strong‑сигнал не превратился в реальную сделку.
- Мы не смогли протестировать стратегию на реальных ордерах, поэтому невозможно оценить её прибыльность.

### Главный вывод:
**Автоматическая торговля — это не только алгоритмы, но и тщательная работа с API биржи, учёт всех фильтров и подготовка тестового окружения.** Первый запуск в production всегда выявляет «детские болезни», которые не заметны при бектестинге.

## 7. Уроки для будущих экспериментов

1. **Всегда тестируйте на демо‑счёте с реалистичными балансами.** Перед запуском пополните счёт тестовыми активами в пропорциях, близких к реальным.
2. **Изучите фильтры символа (LOT_SIZE, NOTIONAL, PRICE_FILTER)** и реализуйте их проверку в коде.
3. **Ведите подробный лог всех действий системы**, включая ордера, балансы и ошибки.
4. **Начните с одной пары**, отладите её полностью, и только потом масштабируйтесь на несколько пар.
5. **Используйте мониторинг** (например, Grafana + Prometheus или простой Flask‑дашборд), чтобы визуально контролировать состояние системы.
