4/12 🚨 Проблема 1: ошибка округления количества (LOT_SIZE)

Binance требует, чтобы количество актива соответствовало шагу stepSize фильтра LOT_SIZE.

Наша система рассчитывала quantity как free_balance * trade_fraction, что давало значения с избыточной точностью (например, 1013.3000000000001).

API Binance возвращал ошибку:
«Parameter 'quantity' has too much precision.»

➖➖➖

5/12 🚨 Проблема 2: недостаточный баланс на демо‑счете

Демо‑счет Binance Testnet изначально содержит лишь небольшое количество тестовых USDT и нулевые балансы по другим активам.

При попытке продажи актива, которого нет на балансе, возникала ошибка:
«Account has insufficient balance for requested action.»

➖➖➖

6/12 🚨 Проблема 3: игнорирование фильтра NOTIONAL

Даже при правильном округлении ордер может быть отклонён из‑за нарушения минимального номинала (minNotional).

Наша система не проверяла этот фильтр для всех пар.

➖➖➖

7/12 🚨 Проблема 4: отсутствие записи ордеров в базу

Хотя ордера размещались через API Binance, их результаты не сохранялись в локальную базу. Это затрудняло анализ P&L.

➖➖➖

8/12 🔧 Как мы исправили ошибки (практические шаги)

• Получаем фильтры символа (LOT_SIZE, NOTIONAL) через exchangeInfo.
• Округляем количество до шага stepSize с учётом нужного количества знаков после запятой.
• Проверяем minNotional и увеличиваем количество, если необходимо.
• Сохраняем каждый размещённый ордер в таблицу orders.

➖➖➖

9/12 📝 Пример кода: округление до шага LOT_SIZE

```python
def _round_to_step(self, quantity, step):
    if step <= 0:
        return quantity
    decimals = int(round(-math.log10(step)))
    return round(math.floor(quantity / step) * step, decimals)
```
